针对你这个“**确定性执行引擎 + 概率性 AI Agent**”的复杂混合系统，命名已经不仅仅是“代码可读性”的问题，而是**防止逻辑灾难**的核心防线。

因为你涉及的概念极多（节点/实例/类型、运行/执行/快照、工具/函数/动作），命名模糊会直接导致后期维护时把`NodeId`当`TypeName`传，或者把`ExecutionId`当`WorkflowId`存。

结合你架构中的核心痛点，给你一套**分层命名哲学**和**具体避坑指南**：

### 1. 核心铁律：必须区分“定义”与“运行时”（消灭歧义）

你的代码里最容易混淆的是“画布上的节点”和“正在跑的节点”。**强烈建议在命名层面强制隔离**：

| 概念      | 推荐命名（定义/设计时）                        | 推荐命名（运行时/实例）                    | 理由                                                |
| :------ | :---------------------------------- | :------------------------------ | :------------------------------------------------ |
| **工作流** | `WorkflowDefinition`                | `WorkflowExecution`             | 杜绝把 `Workflow` 当作参数传来传去。                          |
| **节点**  | `NodeDefinition`（或 `NodeBlueprint`） | `NodeRuntime` 或 `NodeExecution` | 不要用 `NodeInstance`，因为 `Instance` 在 C# 中太重，且有反射歧义。 |
| **连接线** | `Connection`（设计态）                   | `RoutingPath`（运行态）              | 运行态不关心 UI 坐标，只关心数据从哪来。                            |

**行动建议**：你现在的 `NodeInstance` 包含了 `PositionX/Y`（UI信息），又包含 `Parameters`（执行信息）。**拆开**。UI用 `NodeViewModel`，执行用 `NodeDefinition`，数据流用 `NodeExecution`。

***

### 2. 动词严格分级：Trigger / Invoke / Execute / Resume

你的系统有触发（外部）、调度（内部）、执行（节点）、恢复（崩溃重启）四种动作。如果用统一的 `Run`，代码会乱成一锅粥。

- **`Trigger`**：仅用于**外部入口**（Webhook、Schedule）。返回 `TriggerResult`。
- **`Schedule`**：仅用于 Quartz 调度器把 Job 放入队列。不要与 `Execute` 混淆。
- **`Execute`**：**仅用于单个节点**（`NodeRuntime.ExecuteAsync`）。这是原子操作。
- **`Start`**：用于**启动整个工作流**（`WorkflowExecutor.StartAsync`），它会创建 `ExecutionContext`。
- **`Resume`**：**专用于崩溃恢复**（`RecoveryManager.ResumePendingExecutions`）。这个命名能时刻提醒你“恢复时别忘了重建等待区”。

***

### 3. AI / Agent 层的命名“显式化”（防止幻觉）

AI 是最容易“失控”的部分，命名要体现“不确定性”和“循环”。

- **杜绝**：`Process`、`Handle`、`Think`（太虚了）。
- **采用**：
  - **`AgentOrchestrator.IterateAsync`**：强调 Agent 是多轮循环，而不是一次调用。
  - **`ToolRegistry.CollectGadgets`**：建议工具（Tool）不要叫 `Plugin`（太重），统一叫 `Gadget` 或 `Capability`。因为你的工具既有代码片段，又有子工作流，用 `Tool` 容易与 UI 的 `Action` 混淆，用 `Gadget`（小工具）更能体现“即抛”特性。
  - **`FunctionCallingMapper.MapToEngineRequest`**：显式标明这是“LLM幻觉产物”转“引擎实体”的映射层，以后出 bug 了，团队一看这个命名就知道查这里。

***

### 4. 针对“凭据快照”与“等待区恢复”的命名补丁（解决上一轮致命缺陷）

为了让你把上一轮我提到的“快照”和“恢复”落实到位，命名要起得“扎心”一点：

- **凭据**：不要叫 `CredentialService`，要叫 `CredentialSnapshotManager`。
  - 方法：`CaptureSnapshotAsync(executionId)` 和 `UseSnapshot(executionId)`。命名直接告诉你“用的不是热数据，是冰封数据”。
- **等待区**：不要叫 `WaitingArea`（太物理），建议叫 `MultiInputBarrier`（多输入屏障，取自并行计算术语）。
  - 状态属性：不要用 `IsReady`，用 `IsAllRequiredInputsCollected`（自解释）。
  - 恢复方法：`RebuildBarrierFromHistoryAsync(executionId)`。明确告诉后续维护者：“这个函数是为了解决重启丢数据问题的”。

***

### 5. 数据库与 JSON 字段命名规范（强约束）

因为你的系统大量依赖 JSON 序列化存 DSL，字段名必须有强约束：

- **统一使用** **`camelCase`** **序列化**，但 **C# 实体使用** **`PascalCase`**。
- **保留字段标准化**：避免在图数据库和关系库中来回映射。
  - 存储工作流定义的表：`workflow_snapshots`（提示你这东西是版本化的）。
  - 执行记录表：`execution_trails`（不用 `logs`，因为 `logs` 通常指程序输出）。
  - **关联 ID 必须带后缀**：`ParentExecutionId`（不要叫 `ParentId`），`TargetNodeDefinitionId`（不要叫 `TargetId`）。在排查多输入等待问题时，能直接通过命名确认“这连的是定义还是运行时”。

***

### 6. 绝对要避开的“命名地雷”（实测灾难）

在你这套系统里，**严禁**出现以下命名：

- ❌ **`DoWork`**：无数开发者会把业务逻辑塞进这个黑洞，导致 Agent 节点和普通节点无法解耦。
- ❌ **`Model`**：你的系统里有 LLM 模型、数据模型、执行模型。严禁只叫 `Model`，必须叫 `LlmModel` / `DataPayload` / `ExecutionModel`。
- ❌ **`Context`**：你的 `NodeExecutionContext` 会包含 Inputs、Credentials、CancellationToken。严禁缩减为 `Ctx`。必须全拼，因为这是高频使用类。
- ❌ **`Error`**：不要只返回 `Error` 对象。一定要区分 `BusinessError`（业务逻辑错误，如余额不足，可继续）和 `SystemError`（系统崩溃，需重试）。命名不同，错误策略就不同。

### 🔥 针对你核心痛点的“特殊文件夹”解读

之所以这样划分，是为了强行解决我们之前讨论的致命缺陷：

1. **`Snapshots/`** **放在 Runtime 层**：
   - 之前提到“凭据更新导致长事务失败”，这个文件夹专门存放 `ExecutionScopedContext`。它在 `StartAsync` 时创建，在节点 `Execute` 时通过依赖注入提供。名字强制提醒你：**此处数据是只读的“化石”**。
2. **`Barriers/`** **与** **`Recovery/`** **物理隔离**：
   - 不要把“等待区”写在 `Application` 层。它是执行引擎的内部状态机。`Recovery` 文件夹专门放 `RebuildAsync` 方法，它需要同时引用 `Barriers`（重建内存状态）和 `Infrastructure.Persistence`（查数据库历史记录）。这种隔离让后续维护者一眼就能定位“系统启动时做了啥”。
3. **`Orchestrators/`** **与** **`Plugins.AI`** **分离**：
   - `Plugins.AI` 放的是具体的 `Agent 节点`（被引擎调用的“执行单元”）。
   - `Application.Orchestrators` 放的是 `Agent 调度器`（负责调用 LLM、循环迭代、判定是否停止的“大脑”）。
   - 这样设计后，**普通节点**（如 HTTP）只会引用 `Core`，**AI 节点**可以引用 `Orchestrators` 来复用调度逻辑，但 `Orchestrators` 绝不允许反向引用具体的 `Plugins`，防止 Agent 套娃导致的内存爆炸。

***

### 🧵 依赖方向（绝对的“单向铁律”）

为了防止你们的 DLL 引用变成“意大利面条”，必须锁死 `csproj` 的引用关系：

```mermaid
flowchart TD
    Host --> Application
    Host --> Infrastructure
    Host --> Plugins
    
    Application --> Runtime
    Application --> Core
    
    Runtime --> Core
    
    Infrastructure --> Application (实现接口)
    Infrastructure --> Core
    
    Plugins --> Core (只引用Abstractions)
    Plugins -.->|绝不允许| Application
    Plugins -.->|绝不允许| Runtime
```

**为什么 Plugins 不能引用 Application？**
因为插件是“被”引擎调用的（依赖倒置）。如果插件引用 `Application`，它就会尝试去调用 `Orchestrators` 启动新流程，导致 **“执行死锁”**（节点执行到一半想启动另一个执行，嵌套层级失控）。插件只能抛回 `NodeExecutionResult`，由引擎决定下一步。

***

### 📛 命名空间与文件夹严格对应（务必执行）

在 `.NET` 中，最忌讳文件夹是 `Barriers`，命名空间却是 `FlowEngine.Runtime.Waiting`。必须强约束：

- `src/FlowEngine.Runtime/Barriers/MultiInputBarrier.cs` → `namespace FlowEngine.Runtime.Barriers;`
- `src/FlowEngine.Application/Orchestrators/AgentOrchestrator.cs` → `namespace FlowEngine.Application.Orchestrators;`

同时，每个 `Infrastructure` 下的实现类，接口必须来自 `Core` 或 `Application`。例如：
`src/FlowEngine.Infrastructure/Persistence/WorkflowDefinitionRepository.cs` 必须实现 `FlowEngine.Core.Abstractions.Repositories.IWorkflowDefinitionRepository`。

***

### 💡 额外加分项：测试项目的镜像映射

如果你要写单元测试（强烈建议，特别是针对 Recovery 和 Barriers），测试目录结构必须镜像 `src`：

```text
tests/
├── FlowEngine.Core.Tests/         (测试实体行为)
├── FlowEngine.Runtime.Tests/      (⭐ 重点：模拟崩溃恢复、等待区并发)
├── FlowEngine.Application.Tests/  (模拟 Agent 编排逻辑)
└── FlowEngine.Host.IntegrationTests/ (端到端 API 测试)
```

尤其是 `Runtime.Tests`，必须包含一个 `BarrierRecoveryTests.cs`，专门模拟“引擎启动时读取执行记录重建等待区”的场景，验证我们上一轮提到的“数据丢失”不再发生。

***

这种结构下，**“热插拔”**（放 `plugins/` 文件夹）和 **“横向扩展”**（改 `Infrastructure` 连接字符串）会被彻底解耦。你的核心业务逻辑（`Runtime` + `Application`）完全不受外部环境变化影响。
